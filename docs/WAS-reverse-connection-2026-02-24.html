🔄 WAS Reverse Connection

⚡ 핵심 요약

Reverse Connection은 공격자가 방화벽·NAT를 우회하기 위해 피해 서버(WAS)에서 공격자 서버로 먼저 연결을 시작하게 만드는 기법입니다. 일반적인 인바운드 차단 정책을 무력화할 수 있어 탐지가 어렵고, 내부 네트워크 침투 시 주요 수단으로 활용됩니다.

📖 1. 기본 개념: 일반 연결 vs Reverse Connection

일반 연결 (Forward Connection)

공격자 →→→ WAS 서버 (공격자가 서버에 접속 시도)
❌ 방화벽이 외부 인바운드 차단 → 접속 실패

Reverse Connection (역방향 연결)

WAS 서버 →→→ 공격자 서버 (서버가 먼저 아웃바운드 연결)
✓ 아웃바운드(80/443)는 대부분 허용 → 방화벽 우회 성공
💡 핵심 원리: 방화벽은 외부에서 들어오는 인바운드 트래픽은 차단하지만, 내부에서 나가는 아웃바운드 트래픽은 상대적으로 허용하는 경향이 있습니다. Reverse Connection은 이 특성을 이용합니다.
⚙️ 2. WAS 환경에서의 동작 방식

공격은 보통 다음 단계로 진행됩니다:

1
취약점 침투
SQL Injection, 파일 업로드, RCE(원격 코드 실행) 등으로 WAS에 최초 침투
2
악성 코드/웹쉘 업로드
WAS에 웹쉘(JSP, PHP 등) 또는 악성 바이너리를 심음
3
Reverse Shell 실행
WAS가 공격자 C2(Command & Control) 서버에 아웃바운드 연결 개시
4
원격 제어 획득
공격자가 WAS를 완전 제어 — 내부망 横移動(Lateral Movement) 시작
🗂️ 3. 주요 유형

🐚 Reverse Shell 고위험

WAS에서 공격자에게 쉘(bash/sh/cmd) 연결을 열어주는 방식. 공격자가 서버 OS 명령어를 직접 실행 가능. 주로 80/443 포트를 통해 HTTPS로 위장.

🕸️ SSRF + Reverse Connection 고위험

SSRF(Server-Side Request Forgery) 취약점을 이용해 WAS가 내부 시스템이나 공격자 서버에 임의 요청을 보내도록 유도. 내부망 정보 탈취에 활용.

🔄 Reverse Proxy Abuse 중위험

WAS가 Reverse Proxy 역할을 하는 경우, 공격자가 이를 경유점으로 활용해 내부 서버에 무단 접근. 정상 트래픽으로 위장됨.

📡 DNS Rebinding 탐지 어려움

DNS 응답을 조작해 WAS가 공격자 서버를 내부 호스트로 인식하게 만들어 연결을 유도. 브라우저 기반 공격에서 WAS까지 확대 가능.

☁️ 클라우드 환경 특화 주의

AWS/GCP 메타데이터 엔드포인트(169.254.169.254)에 대한 SSRF로 IAM 토큰 탈취. 클라우드 전환 환경에서 급증하는 공격 유형.

⚠️ 4. 위험성 분석
방화벽 우회 가능성매우 높음
탐지 어려움높음
내부망 침투 가능성매우 높음
데이터 유출 위험높음
🚨 핵심 위험 시나리오
WAS가 인터넷과 내부망 사이에 위치한 경우, Reverse Connection을 통해 내부 DB·ERP·HR 시스템까지 침투 경로가 열릴 수 있습니다. 특히 SAP ERP 연동 WAS는 고가치 타겟이 됩니다.

구체적 피해 사례 유형:

  • 내부 DB 서버 무단 접근 → 고객 데이터·영업기밀 유출
  • SAP/ERP 내부 시스템 침투 → 재무 데이터 조작 가능
  • 랜섬웨어 배포 경로로 활용 (내부망 전파)
  • C2 서버로 WAS 활용 → 2차 공격의 발원지로 악용
  • 클라우드 IAM 크리덴셜 탈취 → 인프라 전체 장악
🔍 5. Reverse Connection의 주요 특징

🌐 합법적 포트 사용

HTTP(80), HTTPS(443), DNS(53) 등 정상 트래픽 포트를 사용해 방화벽 룰을 통과. HTTPS 암호화로 내용 검사도 어려움.

🔄 연결 방향이 역방향

피해 서버가 공격자에게 연결을 시작하므로, 방화벽 인바운드 규칙으로는 차단 불가. 아웃바운드 제어가 필수.

👻 정상 트래픽 위장

HTTPS로 암호화되고 일반 웹 요청처럼 보여 IDS/IPS 탐지가 어려움. 비콘 주기를 길게 설정하면 더욱 탐지가 어려움.

🔁 지속적 연결 유지

세션이 끊겨도 주기적으로 재연결(Beacon)하는 구조. 서버 재시작 후에도 지속성(Persistence) 유지 가능.

🛡️ 6. 대응 방안

① 네트워크 레벨

  • WAS의 아웃바운드 트래픽을 화이트리스트 기반으로 최소화 (필요한 도메인/IP만 허용)
  • WAF(Web Application Firewall) 도입으로 비정상 요청 탐지
  • Egress Filtering: WAS에서 나가는 트래픽 전체 모니터링
  • DNS 쿼리 모니터링 — 비정상 외부 DNS 요청 탐지

② 애플리케이션 레벨

  • SSRF 방어: 사용자 입력 URL 검증, 내부 IP 범위(127.0.0.1, 192.168.x.x, 10.x.x.x) 차단
  • 파일 업로드 제한: 실행 가능한 파일 업로드 완전 차단
  • 최소 권한 원칙: WAS 프로세스 계정에 네트워크 접근 권한 최소화
  • 정기적 소스코드 취약점 스캔 (SONAR, OWASP ZAP 등)

③ 탐지/모니터링

  • SIEM 연동: 비정상 아웃바운드 연결 알림 설정
  • EDR(Endpoint Detection & Response) 도입으로 WAS 서버 행위 분석
  • 네트워크 흐름 분석(NetFlow)으로 비정상 통신 패턴 탐지
  • 허니팟/허니토큰: 의심 접근 조기 탐지
⚠️ PerfecTwin/클라우드 환경 적용 시 주의:
오늘 회의에서 논의된 DART IP 오픈 시, 고객사 방화벽에서 허용하는 IP를 최소화하고, WAS ↔ 엔진 간 통신은 mTLS 또는 전용 VPN 채널 사용을 권장합니다. WAF 도입 요구 시 P2P 구조의 제약사항을 명확히 문서화하세요.
