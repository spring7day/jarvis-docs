🔍 SAP Public Cloud ERP — Configuration이란?

SAP S/4HANA Cloud, Public Edition 기준 · 2026-02-26 리서치
출처: eursap.eu · gauravconsulting.com · SAP PRESS Blog
💡 한 줄 요약

SAP Public Cloud에서 Configuration이란 코딩 없이 SAP가 제공하는 Best Practice 범위 내에서 시스템 파라미터를 설정하는 행위입니다. 기존 On-Premise의 Customization(SPRO/ABAP)과는 완전히 다른 개념으로, Clean Core 원칙의 핵심입니다.

⚖️

Configuration vs Customization — 핵심 차이

구분 Configuration ✅ Customization ❌
정의 SAP 표준 파라미터 범위 내 설정 조정 ABAP 코딩으로 SAP 코어 직접 수정
방식 No-code / Low-code
(CBC 도구 사용)		 SPRO, ABAP 개발
(개발자 필요)
Public Cloud ✅ 허용 ❌ 원칙적 불허
업그레이드 영향 없음 (SAP가 관리) 있음 (매 업그레이드 충돌 위험)
대표 예시 회계연도 변형, 통화 설정,
기업구조(조직단위) 정의		 T-code 화면 수정,
커스텀 비즈니스 로직 추가
🛠️

Configuration 실행 도구 — SAP Central Business Configuration (CBC)

1
Scope (스코핑) — 무엇을 쓸지 선택
비즈니스 프로세스의 범위를 결정
  • 국가/지역 선택 → 현지화 자동 적용 (한국 세법, US GAAP 등)
  • Scope Items 선택: FI, SD, MM, PP, Asset Accounting 등
  • 필수 선택 / 선택 선택 / 잠김(티켓 신청) 구분
2
Primary Finance Settings — 재무 기반 설정
기업구조 생성 전 반드시 선행해야 하는 설정
  • 회계연도 변형 (Fiscal Year Variant)
  • 그룹 통화 (Group Currency)
3
Organizational Structure — 기업구조 정의
회사의 법인·운영 단위 설정
  • Company / Company Code 생성
  • Plant, Sales Organization, Distribution Channel, Division
  • Purchasing Organization 등 → CBC가 순서 자동 보장
4
Configuration — 상세 비즈니스 설정
스코프·기업구조 기반의 세부 시스템 설정
  • 가이드된 Configuration Tasks (필수/권장/선택 구분)
  • QAS 배포 → 테스트 → PRD 이관 (Transport Request)
  • Business Process Unit Test 수행 필수
🧹

왜 Configuration인가? — Clean Core 원칙

🚫
Zero Modification
SAP 코어 코드 직접 수정 금지. 표준을 그대로 유지
📦
Standard First
"Why not standard?" 먼저 묻기. 이유 없이 커스텀하지 않기
🔌
API-First Integration
외부 연계는 OData/SOAP API + SAP Integration Suite 사용
☁️
BTP Extension
추가 기능은 SAP BTP에서 Extension으로 개발 (코어 분리)
🧩

Configuration으로 안 되는 건? → Extension 3종

Type 1
Key User Extension (on-stack)
코딩 없이 SAP Fiori 앱에서 UI 필드 추가, 검증 룰 설정 등. 가장 권장
Type 2
Developer Extension (on-stack, ABAP)
Released API/BAdI만 사용. SAP 내부 on-stack에서 ABAP 개발. 제한적 허용
Type 3
Side-by-side Extension (SAP BTP)
코어와 완전 분리. SAP BTP에서 개발. 복잡한 커스텀 요건에 권장
🎯

Clean Core (= Configuration 중심) 채택 시 이점

⚡
자동 업그레이드
반기별 SAP 업그레이드를 코드 충돌 없이 수용
💰
TCO 절감
기술 부채 최소화, 유지보수 비용 감소
🤖
AI 혁신 즉시 접근
Joule AI 등 최신 SAP 혁신을 바로 활용 가능
🔒
시스템 안정성
표준화된 코어로 오류·비호환 위험 최소화
🎭

PerfecTwin 관점의 시사점

SPRO 없음 → Fiori 기반: Public Cloud는 T-code/SPRO 직접 접근이 없거나 제한적. Fiori 앱 기반 UI 테스트 자동화가 필요
Configuration 변경 후 회귀 테스트: CBC에서 Configuration 변경 → QAS 배포 → 비즈니스 프로세스 테스트 자동화가 핵심 니즈
ERP Edition의 적합성: Fiori/웹 UI 기반 시나리오 레코딩·재현 기능이 Public Cloud 테스트에 직접 대응. 기존 WebGUI 기반은 맞지 않음
반기 업그레이드 대응: SAP Public Cloud는 연 2회 자동 업그레이드. 업그레이드 후 비즈니스 프로세스 전체 회귀 테스트 수요가 정기적으로 발생
SaaS 타겟 시장: PerfecTwin의 4월 SaaS 전환 시 Public Cloud 고객이 핵심 타겟. Configuration 이후 테스트 자동화가 주요 가치 제안
